# Use Case: Sprint Planning
## by Voatomy Labs

> **"Replace gut-feel with data-driven estimation"**

---

## Problem
60% of sprint tasks are misestimated. Teams waste 2+ hours in planning meetings debating story points, and 40%+ of sprints experience overruns from gut-feel estimates.

## Solution
ATLAS analyzes six dimensions of complexity to generate sprint plans backed by data — not debate.

## Primary Product
**ATLAS** (#f16e2c) — AI Sprint Planner

## How It Works
1. **Connect** — GitHub, Jira, team calendar
2. **Analyze** — AI maps code complexity, team capacity, design scope, customer demand, tech debt, business priority
3. **Plan** — AI generates sprint plan with confidence scores per ticket
4. **Learn** — Accuracy improves with each sprint cycle (82% → 85% → 87%)

## The 6 Signals
| Signal | What It Analyzes |
|--------|-----------------|
| Code Complexity | Cyclomatic complexity, dependency depth, file coupling |
| Team Capacity | PTO, on-call, skill-task matching, velocity trends |
| Customer Demand | Revenue-weighted feature requests, churn signals |
| Tech Debt | Debt hotspots, velocity impact, remediation overhead |
| Design Scope | Figma complexity, responsive variants, new patterns |
| Business Priority | Pipeline urgency, deal scoring, revenue data |

## Before vs After
| Dimension | Before | After |
|-----------|--------|-------|
| Planning time | 2+ hours | 20 minutes |
| Estimation method | Story point debates | Code-backed estimates |
| Accuracy | ~42% | 87% |
| Risk detection | Mid-sprint surprises | Pre-flagged before start |
| Capacity model | Heads x days | Skills, PTO, on-call factored |
